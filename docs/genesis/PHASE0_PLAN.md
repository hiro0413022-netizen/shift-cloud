## 1. YOZAN Genesis 全体方針の要約

YOZAN Genesisは「AIが会社運営を支援・自動化するCompany AI Operating System」。今回作るのはシフトアプリではなく、その最初の実働モジュールの基盤である。

設計原則:

- マルチテナント（複数会社 > 複数ブランド > 複数店舗）を最初からDBスキーマとRLSに組み込む
- すべての書き込み操作は監査ログを残す
- すべての操作はAPI経由（AIエージェントが人間と同じ操作を実行できる）
- 承認フロー・AI提案の受け皿を最初から持つ
- 将来100モジュールへ拡張可能なモジュール分離構造
- SaaS外販を前提としたテナント分離・課金境界

## 2. YOZAN Workforce OS の位置づけ

Genesis配下の第1ドメインOS。人・勤怠・シフト・給与・人件費を扱う。予約・CRM・在庫・会計などの将来モジュールとは疎結合とし、共通基盤（認証、テナント、権限、監査、通知、AI提案）はGenesis層に置く。Workforce OSはそれを利用する側。

## 3. YOZAN Shift Cloud MVP定義

「シフト・勤怠・給与・店舗運営をひとつに。」

MVPに含む: ログイン / 権限 / 会社・ブランド・店舗・スタッフ管理 / シフトテンプレート / 予定種別 / シフト希望提出（テンプレワンタップ＋メモ） / シフト作成・確定 / 全体・個人シフト閲覧 / iPad打刻 / 勤怠管理・打刻修正 / 月末照合 / 給与見込み・CSV出力 / お知らせ・店舗イベント / AI提案の保存構造 / 監査ログ / 本部ダッシュボード（基本）

MVPに含まない（構造のみ用意）: 顔認証・位置情報打刻 / 予約・CRM・在庫・会計連携 / LINE・Slack通知の実送信 / AI提案の自動生成 / 課金

## 5. DBテーブル一覧

### テナント・組織
- `companies` — 会社（テナントルート）
- `brands` — ブランド（GOLF WING等）
- `stores` — 店舗
- `staff` — スタッフ（auth.usersと1:1、会社所属）
- `staff_store_assignments` — スタッフ×店舗（複数店舗兼務対応、主店舗フラグ）
- `staff_wages` — 時給・給与設定履歴（有効期間つき）

### 権限
- `roles` — ロール定義（システム定義＋会社カスタム）
- `staff_roles` — スタッフ×ロール×スコープ（会社/ブランド/店舗）

### シフト
- `shift_templates` — シフトテンプレート（早番・遅番等、店舗/ブランド単位）
- `schedule_types` — 予定種別（現場・レッスン・有給等）
- `shift_request_periods` — 希望募集期間（対象月・締切）
- `shift_requests` — 日別のシフト希望（テンプレ参照＋メモ）
- `shifts` — シフト（draft/published、確定シフト）

### 勤怠
- `time_records` — 打刻イベント（出勤/退勤/休憩開始/休憩終了、打刻元デバイス）
- `attendance_days` — 日次勤怠集計（実働・遅刻・早退・残業・打刻漏れフラグ）
- `attendance_corrections` — 打刻修正（理由・修正前後・承認者、履歴保存）
- `kiosk_devices` — iPad打刻端末登録（店舗固定・将来のIP/Wi-Fi検証用）

### 給与
- `payroll_periods` — 給与計算期間（月次、締め状態）
- `payroll_items` — スタッフ別給与明細（勤務・残業・交通費・手当・控除・見込額）

### 運営
- `announcements` — お知らせ（会社/ブランド/店舗スコープ）
- `store_events` — 店舗イベント・共有予定
- `notifications` — 通知（ユーザー別、既読管理）

### 基盤
- `audit_logs` — 監査ログ（誰が・何を・いつ・変更前後）
- `ai_suggestions` — AI提案（種別・重要度・対象・推奨アクション・承認/実行状態）
- `approval_requests` — 汎用承認フロー（打刻修正・有給等が使用）
- `integration_configs` — 将来連携の設定受け皿（LINE/Slack/n8n等）

全テーブル共通: `id (uuid)` / `company_id`（テナントキー、RLS基準） / `created_at` / `updated_at` / 論理削除は`deleted_at`。

## 6. 権限一覧

| ロール | スコープ | 主な範囲 |
|--------|---------|---------|
| SaaS管理者 | 全テナント | システム全体（YOZAN内部運用のみ） |
| 会社オーナー | 会社 | 会社内すべて＋給与＋監査ログ |
| 本部 | 会社 | 全店舗の運営・比較・AI提案 |
| エリアマネージャー | 複数店舗 | 担当店舗の管理・シフト・勤怠 |
| 店舗責任者 | 店舗 | 自店舗の管理・シフト作成・打刻修正 |
| 社員 / アルバイト / 業務委託 / レッスンプロ / スタッフ | 本人＋所属店舗 | 自分のシフト・希望提出・全体シフト閲覧（給与は本人分のみ） |
| 閲覧専用 | 付与スコープ | 読み取りのみ |

実装: ロール＝権限セット（permission flags）。雇用形態（社員/アルバイト等）はロールと分離し`staff.employment_type`として持ち、給与計算ルールに使用。給与画面はロールに加え追加認証（再認証 or PIN）を要求。

## 8. クレジット消費を抑える開発運用

指示された12ルールを採用。運用ファイル:

- `/docs/genesis/DECISIONS.md` — 設計判断を1行ずつ追記。同じ議論を繰り返さない
- `/CHANGELOG.md` — 実装後は差分のみ記録
- `/NEXT_TASKS.md` — 次回作業の対象ファイルを事前に限定
- 1作業＝1画面 or 1機能。実装前に対象ファイルを宣言してから着手
- 共通ルールは docs/genesis、Workforce固有は docs/modules/workforce-os を参照し再生成しない

## 9. 実装前の確認事項

1. **Supabase**: 新規プロジェクトを作成してよいか（Supabase連携は接続済み。無料/有料プランどちらか）
2. **スタッフのログイン方法**: アルバイトはメールアドレスを持たない可能性 → メール＋パスワードを基本とし、メールなしスタッフは「ログインID＋パスワード」方式を併用でよいか
3. **リポジトリ構成**: MVPは単一Next.jsアプリ＋内部パッケージ分離（モノレポは形だけ用意）を推奨。Turborepoフル構成は店舗数拡大時に移行でよいか
4. **勤怠の丸めルール**: 実働は1分単位で記録し、給与計算時の丸め（15分単位等）は会社設定でよいか。深夜割増・残業割増率の初期値は？
5. **言語**: UIは日本語のみでよいか
6. **GitHub/Vercel**: リポジトリ作成先とデプロイ先の指定はあるか
