1. Cowork または Claude Code を **YOZAN GENESIS リポジトリに接続**した状態で開く

# YOZAN GENESIS アップグレード指令

あなたはYOZAN GENESISの開発を引き継ぐシニアアーキテクトです。
このリポジトリは10日間で11アプリを量産した高速プロトタイプ群であり、**今から「量産できる基盤」へ作り替える**フェーズに入ります。

## 0. まず読むもの（正典。ここに書かれた決定を勝手に覆さない）

- `docs/genesis/VISION.md` — North Star。何を作っているか
- `docs/genesis/DECISIONS.md` — 34件の決定ログ。**同じ議論を繰り返さない**
- `docs/genesis/OPERATIONS.md` — ユーザー（古川さん）作業の正典
- `docs/genesis/AI_RULES.md` / `DEVELOPMENT_RULES.md` / `DATABASE_STANDARD.md` / `API_STANDARD.md` / `SECURITY.md`
- `NEXT_TASKS.md` / `CHANGELOG.md`

## 1. 絶対に守る制約（違反したら差し戻し）

- **push・デプロイ・本番DB変更・課金・外部送信は実行しない。** 提案と準備まで。実行は古川さんの承認後（VISION §7）
- **Publicリポジトリ（DECISIONS #14）。** 秘密情報・パスワード・トークンをコードやdocsに一切書かない。秘密は `vault_systems`（DB）のみ（#26）
- **クレジットを浪費しない。** 全ファイル読み込みのような総当たりをせず、狙って読む。長い出力を無意味に繰り返さない
- **既存DECISIONSを覆す提案は「#N を更新する提案」として明示**し、理由を書く。黙って変えない
- **マイグレーションは追加のみ。** 既存テーブルの破壊的変更をしない
- 金額は integer 円、時間は integer 分（#4）。論理削除（#5）

## 2. 現状の既知の弱み（これを潰すのが仕事）

1. **テストがゼロ。** 給与計算・売上集計・月会費予測という**金額に直結するロジック**が手動確認だけで守られている
2. **数字が信用しきれない。** 月会費は予測値（#31/#32）、6月のゴルフ経費は未入力で**利益が過大**。CEO AIはこの数字を読んで経営判断する設計なので、土台が揺れている
3. **コード重複。** `auth.ts` / `kernel.ts` / `supabase/` が7〜8アプリにコピーされている。`packages/` は **ARCHITECTURE.md に書かれているのに実在しない**（#10で保留したまま、アプリが11本になった）
4. **出す速度が作る速度に追いつかない。** member-os / survey-os / reserve-os が実装済み・未デプロイ
5. **管理の綻び。** マイグレーション番号 0024 が2本、DECISIONS の #28/#29 が重複採番、`sales-support-saas/` が `apps/` の外にある
6. **Bus factor = 1。** 古川さん＋AIのみ。docsとDECISIONSが命綱

## 3. あなたのゴール（優先順位つき）

### 最優先: 開発効率 — 「12本目のアプリを1日で出せる」状態を作る
今の量産パターン（入力面は独立アプリ / GENESISは閲覧＋承認 / DBは共有）は勝ちパターン。**これを型として固定化**する。

- `packages/` を実際に作り、`auth` / `kernel` / `supabase client` / `ui` / `config` を集約（#10 の「2アプリ目でpackages化」を、11アプリ目にしてようやく履行する）
- **新規アプリのスキャフォールド**（`scripts/new-app.ts` 等）: アプリ名とスキーマ接頭辞（`svy_` `res_` 等）を渡すと、ログイン・認可・レイアウト・`/api/v1`・Vercel設定の雛形が生成される
- `docs/genesis/MODULE_TEMPLATE.md` を、その生成器と一致する内容に更新
- **デプロイ手順を1本のチェックリストに**（OPERATIONS.md）。「ユーザー作業(1)(2)(3)…」が毎回散らばっている現状を、コピペで終わる定型に

### 次点: 検証 — 「間違った数字でCEO AIが判断する」事故を止める
- **金額ロジックのテストを書く**（給与計算、売上集計、月会費予測 `refresh_golfwing_membership_forecast`、KPI集計関数）。まずここだけでいい。網羅率は追わない
- **KPI整合性チェッカー**を日次cron（`runDailyCeoReport`）に組み込む。Claude API不要、SQL＋ルールで十分:
  - 経費が0円の月がある / 売上が前月比±50%超 / 予測値のまま実績が入っていない月 / KPI目標値が未設定
  - 検知したら CEO AI の**判断リストに上げる**（例: 「6月のゴルフ経費が未入力です。利益が過大に見えています」）
- CI（`.github/workflows`）で `next build` ＋ テスト ＋ `tsc --noEmit` を全アプリに走らせる

### 3番目: 現場が使えるか — 実際に働く人間の手順
GENESISは古川さん専用だが、**Shift Cloud / member-os / Legal OS / Reserve OS はアルバイト・スタッフが触る**。
- 各アプリの**現場向け手順書**（`docs/modules/*/RUNBOOK.md`）を作る。「タブレットで受付する」「打刻を修正する」「契約書を登録する」を、PCに詳しくない人が読んで完了できるレベルで
- スタッフが**詰まる箇所**をコードから洗い出す（エラーメッセージが不親切 / 権限不足時に何も表示されない / 保存に失敗しても気づけない、等）
- 権限（`use_reception` / `use_legal` / `use_survey` / `view_hq`）の付与手順が OPERATIONS.md に無ければ追記

### 4番目: 掃除
- マイグレーション番号重複（0024×2）の解消、DECISIONS の重複採番（#28×2, #29×2）を採番し直し
- `ARCHITECTURE.md` を**実態に合わせて書き直す**（packages/が無い、アプリが11本、独立アプリ方式が確立、を反映）
- `sales-support-saas/` を `apps/` 配下へ、または「別物である」と明記

## 4. 進め方（これを厳守）

### Phase 0: 監査 — **ここまでやって必ず止まる**
コードとdocsを読み、以下を出力する。**この時点でファイルを変更しない。**

1. **改善バックログ**: 各項目に `影響度(高/中/低)` × `工数(時間)` × `AIだけで完結するか / ユーザー作業が要るか` を付ける
2. **上位5件の推奨順序**と、その理由
3. **危険な発見**（金額が間違って計算されうる箇所、権限の穴、データ欠損）を最優先で報告
4. **DECISIONS更新提案**があれば `#N を更新: 理由` の形で列挙

出力は簡潔に。長大なコード引用をしない。

### Phase 1以降: 古川さんが選んだ項目のみ着手
- 1項目ずつ。**着手前に「変更するファイル一覧」と「壊れうるもの」を先に提示**して確認を取る
- 変更後は `next build` と `tsc --noEmit` を該当アプリで通す
- **commitまで。pushはしない**（古川さんのPCから行う）
- 完了したら `CHANGELOG.md` / `NEXT_TASKS.md` / 必要なら `DECISIONS.md` を更新する

## 5. 完了の定義

「12本目のアプリを、古川さんが1日で本番に出せる」— これが達成できたら成功。
