## 1. 危険な発見（最優先）

### 🔴 D-1: 給与計算の月末日バグ（buildPayroll）
`apps/shift-cloud/src/app/admin/payroll/actions.ts` が期間上限を `${ym}-31` 固定で生成。31日が無い月（6月・9月等）は `attendance_days.date <= '2026-06-31'` が **Postgresのdate型キャストエラー** になり、勤怠0件扱い→「対象月の勤怠データがありません」で給与計算が失敗する可能性が高い。
→ **本監査に続くPhase 1で修正＋テスト固定済み**（payroll-calc.ts の monthRange）。

### 🟠 D-2: 丸めと残業の相互作用で基本給が過少になり得る
日次 `work_minutes` を丸め（floor）た合計から **丸めていない** `overtime_minutes` を引いて通常分を出すため、丸め幅＞残業の日があると通常分が実際より小さく（理論上は負にも）なる。→ Phase 1で `Math.max(0, ...)` ガード＋テスト固定。丸め仕様自体はDECISIONS #9の会社設定のまま。

### 🟠 D-3: 時給の月中変更が月全体に遡及
`wageFor` は「月末時点で有効な最新賃金」を月の全日に適用。月中の時給変更は変更前の日にも新時給が適用される。現状GOLF WINGの運用（月初改定）なら実害なしだが、仕様として明記されていない。→ バックログ（B-7）。

### 🟠 D-4: 6月ゴルフ経費未入力→利益過大（既知 #31）
CEO AIがこの数字で判断する構造的リスク。→ Phase 1で **KPI整合性チェッカー**を日次cronに組込（経費0円月／forecast残存月／売上前月比±50%超／KPI目標未設定を検知して判断リストへ）。

### 🟡 D-5: 銀行CSV科目推測（categorize.ts）の振込→人件費判定
法人マーカーの無い振込を一律 `labor` に分類。誤分類は月次PL→人件費率KPIへ波及。コメントに「経理AI置換予定の暫定」と明記済み・取込時に人が確認する運用なので中リスク。→ バックログ（B-8）。

### 🟡 D-6: 検証手段の不在
テスト0件・CI無し。さらにこの開発環境ではローカルtsc/next buildが信頼できない（マウント同期・マルチバイト誤読、memory記録済み）。**GitHub Actions CIが唯一の恒久的な検証手段** → Phase 1で導入。

## 3. コード重複の実態

`auth.ts` / `kernel.ts` / `supabase/` は7〜8アプリにコピーされ、**全て内容が微妙に分岐済み**（md5全部不一致）。よって「既存アプリの一括packages移行」は回帰リスクが高い。方針:

1. **新規アプリから** `packages/core` を使う（scaffold生成器とセット）＝12本目を1日で出す道
2. 既存アプリの移行は1アプリずつ・デプロイ検証付きで後続（バックログB-6）
